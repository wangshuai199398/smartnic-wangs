RDMA 核心概念速览

概念含义硬件中的角色
QP (Queue Pair)RDMA 通信通道，含发送队列 SQ 和接收队列 RQ保存通信状态、队列地址和索引
CQ (Completion Queue)完成通知队列硬件写完 CQE 后通知软件
MR (Memory Region)注册的网卡可访问内存区域权限检查、地址转换、引用计数
PD (Protection Domain)资源隔离边界防止 QP 访问不属于自己的 MR
Doorbell软件通知硬件的快速通道通过 BAR0 MMIO 写入触发
WQE (Work Queue Entry)软件提交给硬件的工作描述64 字节 packed struct，含操作类型/地址/key
CQE (Completion Queue Entry)硬件返回给软件的完成描述64 字节 packed struct，含状态/字节数/WR ID
DMA硬件直接读写主机内存受 MR 保护，含 SGE 遍历和分块传输
RoCEv2Ethernet/IP/UDP 承载 RDMA将 RDMA 操作封装成网络包

软件写 WQE → Doorbell → SQ Engine → WQE Fetch → DMA Dispatcher → SGE Traversal → MR Check → Host Read → Transport → CQE Write → MSI-X

控制面流程

软件写 BAR2 CSR → CSR Mailbox → Lifecycle Manager → Context Table → Done Response

3 smartnic_pkg.sv —— 核心数据包

smartnic_pkg.sv 是整个项目的唯一共享定义文件，1389 行，被所有模块通过 import smartnic_pkg::* 引用。它不包含任何逻辑，只定义类型和常量。

💡 为什么把所有类型集中在一个 package 里？
  • 单一事实来源：RTL、Cocotb 测试、未来 Linux 驱动 UAPI、用户态库全部引用同一套类型定义，消除不一致。
  • 编译时类型检查：SystemVerilog packed struct + enum 确保位宽精确匹配，防止跨模块连线错误。
  • 易于演进：添加新字段只需改一处，所有模块自动获得更新。

4 PCIe 子系统 (5 模块)

PCIe 子系统是 SmartNIC 与主机之间的物理接口层。它将来自 PCIe 总线的 TLP (Transaction Layer Packet) 请求转换为内部模块可以理解的 BAR 访问。

🔌 pcie_endpoint_wrapper

PCIe 硬核 IP 的接口封装层。将特定 FPGA 厂商的 PCIe IP 接口适配为统一的内部 valid-ready 接口，是外部 IP 与内部逻辑的隔离边界。
🗺️ pcie_bar_decoder
rtl/pcie/pcie_bar_decoder.sv
根据 BAR 编号和 offset 将内存读写请求路由到 Doorbell (BAR0)、CSR (BAR2) 或 MSI-X (BAR4) 子路径。含对齐和范围检查。
⚙️ pcie_cfg_space
rtl/pcie/pcie_cfg_space.sv
PCIe Type 0 配置空间。提供 Vendor ID、Device ID、BAR 属性、Capability 链表 (PM, MSI-X, PCIe, AER, ATS, SR-IOV) 等。
🔐 pcie_function_manager
rtl/pcie/pcie_function_manager.sv
SR-IOV PF/VF 身份管理和资源窗口分配。每个 function 有独立的 QP/CQ/MR/Doorbell/MSI-X 资源窗口，实现硬件隔离。
📬 pcie_msix
rtl/pcie/pcie_msix.sv
MSI-X 中断表/PBA 管理和中断调节 (moderation)。支持 8 个 vector，含 per-vector mask、moderation timer/counter。
pcie_bar_decoder
rtl/pcie/pcie_bar_decoder.sv

纯组合逻辑路由 + 一拍寄存响应的 0-cycle 延迟解码器。不使用 FSM，所有输出在同周期确定。

💡 为什么 BAR decoder 不用 FSM？

BAR 路由是纯地址解码——输入 offset 和 BAR 编号立刻决定目标。用组合逻辑可以做到 0 周期延迟。唯一的寄存器是 bar_rsp_valid/bar_rsp_rdata，用于给上游返回响应。

5 Doorbell 子系统 (5 模块)

Doorbell 是软件通知硬件的快速路径。用户态程序通过 MMIO 写 BAR0 Doorbell page 来告知硬件：SQ/RQ 有新 WQE，或 CQ 已被软件消费。

🔔 doorbell_decoder
rtl/doorbell/doorbell_decoder.sv
根据 BAR0 offset 解析 Doorbell 类型 (SQ/RQ/CQ_ARM)，从 page_id 推导 QPN/CQN，提取 queue index。
🛡️ doorbell_access_check
rtl/doorbell/doorbell_access_check.sv
调用 pcie_function_manager 校验 Doorbell 发起者是否有权限访问目标 QP/CQ。
📤 sq_doorbell_handler
rtl/doorbell/sq_doorbell_handler.sv
处理 SQ Doorbell：更新 QP context 中的 sq_producer index，触发 SQ engine 开始处理新 WQE。
📥 rq_doorbell_handler
rtl/doorbell/rq_doorbell_handler.sv
处理 RQ Doorbell：更新 QP context 中的 rq_producer index，使能接收。
🔄 cq_arm_doorbell_handler
rtl/doorbell/cq_arm_doorbell_handler.sv
处理 CQ Arm Doorbell：更新 CQ consumer index，重新 arm 中断通知。
doorbell_decoder
rtl/doorbell/doorbell_decoder.sv

Doorbell 的关键设计是 Page-based 寻址：BAR0 被划分为多个 4KB page，每个 page 对应一个 QP 或 CQ。Page 内的 offset 区分 SQ (offset 0)、RQ (offset 8) 和 CQ Arm (offset 16)。

内部信号与函数

信号/函数类型功能
page_offset组合逻辑= db_req_offset & (DB_PAGE_SIZE - 1)，获取 page 内偏移
page_id组合逻辑= db_req_offset >> DB_PAGE_SHIFT，推导 QPN 或 CQN
decoded_type组合逻辑根据 page_offset 匹配 DB_SQ_OFFSET / DB_RQ_OFFSET / DB_CQ_ARM_OFFSET
decoded_type_valid组合逻辑是否为已知的 Doorbell 类型
decode_status_next组合逻辑综合 page_in_range、dword_aligned、write_strobe_valid、decoded_type_valid 给出最终状态
sq_doorbell_payload_tpacked structSQ Doorbell 的 payload 格式：flags + doorbell_sequence + new_sq_producer_index
rq_doorbell_payload_tpacked structRQ Doorbell 的 payload 格式
cq_arm_doorbell_payload_tpacked structCQ Arm Doorbell 的 payload 格式
💡 为什么用 Page-based 寻址？
  • 用户态直接访问：4KB page 可通过 mmap 映射给用户进程，避免系统调用。
  • 硬件解码简单：page_id 直接就是 QPN/CQN，无需查表。
  • 安全隔离：每个 VF 有独立的 BAR0 window，doorbell_access_check 验证跨 VF 访问。

6 QP 子系统 (6 模块)

QP (Queue Pair) 是 RDMA 通信的核心对象。QP 子系统负责 QP 的创建/修改/查询/销毁、状态迁移校验、SQ/RQ 引擎的数据处理，以及 QP 销毁/错误时的清理。

📋 qp_context_table

🔄 qp_lifecycle_manager

处理 CREATE_QP / MODIFY_QP / QUERY_QP / DESTROY_QP / QP_TO_ERROR 命令。7 状态 FSM。
✅ qp_state_validator
rtl/qp/qp_state_validator.sv
纯组合逻辑校验 IBTA 规范的 QP 状态迁移是否合法。检查必需属性是否齐备。
🧹 qp_cleanup_manager
rtl/qp/qp_cleanup_manager.sv
DESTROY_QP / QP_TO_ERROR 的清理流程：阻止新 Doorbell、等待 in-flight 完成、为未消费 WQE 生成 flush completion。
📤 sq_engine
rtl/qp/sq_engine.sv
Send Queue 引擎：收到 Doorbell 后查找 QP、取 WQE、解码 opcode、分发到 DMA/transport。
📥 rq_engine
rtl/qp/rq_engine.sv
Receive Queue 引擎：入站 Send 到达后查找 QP、取 Recv WQE、分发 DMA write。

💡 为什么 MR 表需要引用计数 (refcount)？

当一个 MR 正在被 DMA 读取时，如果软件同时发起 DEREGISTER_MR，硬件不能立即删除 MR entry——否则正在进行的 DMA 会访问到无效的物理地址。引用计数确保：

  • 每次 DMA 发起前 refcount +1
  • 每次 DMA 完成后 refcount -1
  • DEREGISTER_MR 标记 pending_deregister=1，然后等待 refcount 归零后才真正清除表项
  • refcount 归零超时（MR_DEREG_TIMEOUT_CYCLES=1024 周期）会报错，防止死锁
MR 保护管线 (dma_mr_integration 中串联)
rtl/dma/dma_mr_integration.sv

DMA 数据通路中的 MR 保护是一个多级串联管线，每级独立可测试：

dma_segment → ACCEPT (锁存) → KEY_CHECK (mr_key_checker) → ACCESS_CHECK (mr_access_checker) → PD_CHECK (mr_pd_checker) → TRANSLATE (VA→PA) → REFCOUNT_INC (+1) → EMIT (protected_segment)

每级失败都会产生细粒度的 dma_mr_error_e 错误码（15 种），方便驱动定位问题根因。

💡 为什么保护管线分这么多级，而不是合并成一个模块？
  • 单一职责：key_check 只管 key 方向，access_check 只管权限位，pd_check 只管 PD 匹配。各自独立，修改一处不影响其他。
  • 独立测试：每个 checker 可以单独用 Cocotb 穷举测试所有正常和错误路径。
  • 流水线并行：在 ASIC 实现中，每级可以独立作为 pipeline stage，提高吞吐。
  • 错误归因精确：driver 可以根据 error code 立刻知道是 key 错误、权限错误还是 PD 错误，而不是一个笼统的 "MR error"。

SGE Traversal 是将 SGE list 转换为规范化的 DMA segment 流的关键模块。它检查：

  • Index 单调性：SGE index 必须严格递增，不允许重复或乱序

分块读取策略

对于超过 DMA_MAX_READ_BYTES (32 字节) 的 segment，自动拆分为多个 PCIe read chunk。每个 chunk 携带 dma_read_tag_t (desc_id + segment_index + chunk_index) 用于匹配响应。

💡 为什么需要 RELEASE_REF 状态？

DMA 从 MR 读取数据完成后，必须释放之前 MR integration 阶段增加的 refcount。如果忘记释放，MR 永远无法 deregister，造成资源泄漏。将 refcount release 作为独立 FSM 状态确保了：

  • 即使 payload emit 成功，也不会跳过 refcount release
  • 即使 payload emit 失败（下游 backpressure），也会在错误路径中释放 refcount
  • ref_token 在 protected_segment 中携带，传递完整上下文

11 整体设计思想与原则

11.1 Valid-Ready 握手 —— 全模块统一接口

所有模块间的数据交换都使用标准的 valid / ready 握手协议：

  • 生产者 拉高 valid 表示数据有效
  • 消费者 拉高 ready 表示可以接收
  • valid && ready 同时为高时，数据传输发生（一拍握手）
  • 这种协议天然支持反压（backpressure）：消费者不 ready 时生产者必须保持 valid 和数据不变
💡 为什么全部使用 valid-ready 而不是 AXI-Stream？
  • valid-ready 是最小化的流控协议，不需要 AXI-Stream 的 tlast/tkeep/tid/tdest 等附加信号。
  • 每个模块的接口信号在 packed struct 中已定义好，valid-ready 只是传输控制。
  • 后续如果需要升级到 AXI-Stream，只需加 thin wrapper 把 valid/ready 映射到 tvalid/tready。

11.2 每模块一个 FSM —— 可测试性原则

每个模块都实现为独立的有限状态机（FSM），包含：

  • 状态枚举：*_state_e，定义在 smartnic_pkg.sv 中
  • 状态寄存器：state_reg
  • always_ff 块：时钟上升沿更新状态
  • always_comb 块：组合逻辑计算下一个状态和输出
  • 错误枚举：*_error_e，每种失败原因都有独立编码

11.3 单一 Package 文件 —— 类型集中管理

smartnic_pkg.sv 是整个项目的类型定义中心：

  • 所有 bit-width parameter 集中定义，方便统一调整
  • 所有 packed struct 精确指定位宽，防止综合器推断出不期望的位宽
  • 所有 enum 定义在同一文件，RTL 和 Python testbench 都用同一套常量
  • Package 不包含任何可综合逻辑，只包含 parameter、typedef、localparam

11.4 控制面与数据面分离

层面入口速率模块
控制面BAR2 CSR Mailbox低（管理操作）csr_mailbox → lifecycle_manager / registration_manager → context tables
数据面BAR0 Doorbell高（每包触发）doorbell_decoder → sq_engine/rq_engine → DMA → completion

两条路径共享 context tables (QP/CQ/MR)，但通过 SR-IOV resource window 和 owner_function 字段确保隔离。

11.5 原型优先，正确性优先

  • 寄存器阵列而非 CAM：1024 项的 context table 用寄存器 + 组合逻辑遍历实现，适合 FPGA 原型，ASIC 可替换为 CAM/TCAM
  • 固定优先级仲裁：DMA dispatcher 用固定优先级而非 DRR，先保证正确性
  • 分块 DMA：单次 PCIe read 最大 32 字节（DMA_MAX_READ_BYTES），简化 PCIe IP 接口要求
  • 超时保护：所有等待状态都有超时计数器（deregister 等 refcount、cleanup 等 in-flight、traversal 等 last SGE）

11.6 细粒度错误报告 —— 可调试性

每个模块都定义了 10-16 种独立的错误码（16-bit wide），使得：

  • 驱动可以精确知道失败原因并打印有意义的错误信息
  • 仿真测试可以注入特定错误条件并验证对应错误码
  • 硬件调试时，错误码可以快速定位 bug

11.7 模块级测试驱动

Cocotb Makefile 为每个模块定义了独立的测试目标：

# sim/cocotb/Makefile 中的典型测试目标 test-pcie-cfg: // pcie_cfg_space 单元测试 test-doorbell-decoder: // doorbell_decoder 单元测试 test-qp-lifecycle: // qp_lifecycle_manager 单元测试 test-mr-registration: // mr_registration_manager 单元测试 test-dma-host-read-path:// dma_host_read_path 单元测试

11.8 完整的数据结构层次

smartnic_pkg.sv (类型定义根)
  ├── parameter (位宽、容量、BAR布局) ── 编译期常量
